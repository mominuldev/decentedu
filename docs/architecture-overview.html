Executive summary

Safe Eduman is a production school/college management ERP for the Bangladeshi education system, sold by Safe IT Ltd and serving multiple school branches under one account. The login page hints at four features; the authenticated app holds twelve full modules — from an admissions pipeline and biometric attendance to a deep examination and results engine, student billing, double-entry accounting, an SMS gateway and a public-website CMS.

The rebuild preserves every confirmed business workflow while replacing the dated Blade/Livewire UI with a React + Tailwind single-page app over a versioned Laravel REST API. The backend is a modular monolith — one database, one deploy, ACID transactions — because the domain is densely relational (a marksheet touches student, enrolment, class, subject, grade scale and branch in one request). Modules give clean domain boundaries without distributed-systems cost.

The existing system

What the crawl actually found — the basis for everything that follows.

Stack & shape

Laravel + Livewire + AdminLTE 3 (Bootstrap 4), jQuery, Select2. Server-rendered Blade — not an API/SPA today. Session-cookie auth (safe_eduman_session, XSRF-TOKEN). Bilingual EN/বাংলা (name_bn columns). Almost certainly MySQL.

Everything is scoped by two things

Active branch and active academic session frame every screen — they are ambient context, not per-form parameters. The rebuild models both as first-class request context enforced centrally.

The demo’s 500-error pages are not missing features

Several legacy routes returned server errors on the demo instance because that branch’s data isn’t configured — but the routes, menus and configuration screens prove the features exist. They are treated as confirmed and reproduced.

04

The data backbone

The single most important modelling decision, drawn straight from the app’s own structure.

class_config — a concrete teaching unit of Class × Shift × Section (the app labels them “Six-A-Day”) — is the hub that attendance, routines, exam scope and fee config all reference. A separate student_enrollments table holds each student’s placement per session, so promotion history, past results and re-admission are all correct rather than overwritten.

students
Permanent identity — name, guardians, contact, status. Never carries a class.
student_enrollments
Per-session placement: academic_year, class_config, group, roll, is_current. History lives here.
class_configs
Class × Shift × Section. The shared hub for attendance, routine, marks & fees.

On top of this sit the confirmed exam rules: exam types Weekly · Monthly · Final · Grand Final roll up through a Final Mark Config, grades are scaled per class, and merit is ranked by one of four confirmed methods (Total-Mark / Grade-Point × Sequential / Non-Sequential).

05

Multi-branch & access control

Single database, row-level isolation, branch-scoped roles — matching the observed “one account, many schools” model.

Tenancy — Organization → Branch

Every business row carries a branch_id. Isolation is enforced three ways: middleware asserts branch membership, a global Eloquent scope auto-filters and stamps every query, and policies re-check the record’s branch on binding. A user guessing another branch’s record id gets a 404. Cross-branch reports need an explicit report.cross_branch permission.

RBAC — role + grant + branch

Built on spatie/permission with the teams feature (team_id = branch_id) for branch-scoped roles, plus per-user grants for the “Permissions / Tools” overrides the app exposes. Permissions are module.resource.action, shared by menu, API and UI.

Confirmed roles: Super Admin, Admin. A wider set (Accountant, Exam Controller, Teacher, Admission Officer, Viewer) is proposed and clearly marked as pending your approval — none will be seeded without sign-off.

Key decisions & why

DecisionChoiceWhy
App styleModular monolith + REST API + React SPA Densely relational, transactional domain; one DB & ACID beat distributed complexity at this scale. Modules keep boundaries; extractable later.
Multi-branchSingle DB, branch_id scoping Shared reference data & cross-branch reporting are first-class; row-level isolation is simpler and cheaper than DB-per-tenant.
Authorizationspatie/permission, branch-scoped teams + policies Matches the observed role + granular-permission + branch model exactly; mature, cached checks.
AuthSanctum SPA cookie sessions First-party SPA; no token in JS → no XSS theft; CSRF-protected.
Server stateTanStack Query + small Zustand store ~95% of state is server data needing caching & invalidation; avoids Redux boilerplate.
Data accessEloquent + Query objects (no generic repos) Eloquent is the data layer; Query objects standardise list filtering without ceremony.
Heavy workRedis queues via Horizon Result processing, PDF reports, bulk imports, SMS & device ingestion must be async to keep requests fast.
ReportingQueued server PDF + Laravel Excel Marksheets, tabulation & admit cards are print-critical; large jobs can’t block requests.

08

Principal risks

  • High
    Exam & result rules are the deepest hidden logic. GPA, 4th-subject, merit methods and term weighting. Mitigation: extract exact rules from legacy code/DB, unit-test each, and run a parity harness against known-good legacy outputs.
  • High
    Branch isolation must be provably correct. A leak is a serious breach. Mitigation: triple-layer enforcement plus a dedicated isolation test suite gating every release.
  • High
    Legacy → new data migration & cutover. Mitigation: importers validated against the field inventory, staged dry-runs, per-branch cutover with rollback.
  • Med
    Livewire → React interaction change & bulk grids at scale. Mitigation: preserve workflows, virtualized editable grids with batch save and partial-failure reporting, a usability pass.
  • Med
    Financial & SMS integrity (debit = credit, paid SMS, external delivery). Mitigation: DB + application invariants, audit log, balance guardrails, rate limits, idempotency.

Open questions — the approval gate

Per the engagement’s own rule, no implementation, migrations or components have been written. These resolve before Phase 1.

Access & parity · highest leverage
  1. Can you share the legacy source code and/or a DB dump? It replaces the last inferred rules (exact GPA / fine / status logic) and enables accurate data migration.
  2. Confirm DB engine (assumed MySQL) and hosting constraints — affects Redis, queues and object storage.
Product scope
  1. Roles: keep the minimal Super Admin / Admin + granular permissions, or seed the proposed role set (Accountant, Exam Controller, Teacher, Admission Officer, Viewer)?
  2. Teacher / guardian portals and online fee payment (bKash / Nagad / SSLCommerz) — v1 or future?
  3. Single operator managing all branches, or multi-organization self-serve SaaS?
Domain specifics to confirm
  1. Exact term-weighting into Final / Grand Final, 4th-subject GPA treatment and merit tie-breaking.
  2. Fee & fine rules (flat vs per-day, partial payments, waiver stacking) and fees → GL mapping.
  3. Biometric device protocol for the ingestion adapter; student status lifecycle & re-admission.
